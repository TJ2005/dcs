Project Overview

What We Are Building

Sentinel is a lightweight LLM security middleware that intercepts, analyzes, and secures every interaction between users and any Large Language Model. It acts as a drop-in proxy — any application pointing at Sentinel gets full security coverage without changing a single line of application code.
 
The 4-Pillar Architecture

Note: Pillars 1 and 2 from the original spec have been merged — Input Analysis and Ensemble Detection are a single pipeline, not separate pillars. This is more accurate and easier to explain to judges.
 
Pillar	Component	Purpose

1	Detection Pipeline	Intercepts every incoming message. Runs rule engine, GLM triage, and Qwen deep analysis in sequence. Returns a structured ThreatEvent with score, category, reason, and layer_triggered.

2	Adaptive Policy Engine	Enforces natural-language policies written by admins. Hot-reloadable with zero downtime. Now respects Firewall Mode setting.

3	Output Scrubber	Audits every LLM response for PII, secrets, system prompt echo, and harmful content using Nemotron 9B + Presidio. Fails closed.

4	Threat Dashboard	Real-time visualization with live SSE feed, attack charts, Policy Studio, Firewall Mode toggle, and User Trust controls.
 
Tech Stack

Layer	Technology	Notes

Middleware proxy	FastAPI + uvicorn	Async Python, OpenAI-compatible API surface

Fast triage model	GLM 4.5 Air (free)	Non-thinking mode, 300ms hard timeout, OpenRouter

Deep analysis model	Qwen3 480B A35B (free)	Semantic + policy enforcement, OpenRouter

Output auditor	Nemotron 9B V2 (free)	PII + secret detection, fails closed if timeout

Frontend dashboard	React (Vite) + Recharts	SSE live feed, charts, Policy Studio, Firewall Mode

Key Technical Decisions

Threat Scoring Formula

The ensemble produces a final threat score from 0–100 by combining three detection layers, then applying a session multiplier, and then capping by Firewall Mode.
 
Layer	Weight	When called	Latency

Rule-based engine	15%	Always. Regex, base64, unicode homoglyphs.	~5ms

GLM 4.5 Air (triage)	30%	Always. Fast semantic check, non-thinking mode.	~200ms (300ms hard limit)

Qwen3 480B (deep)	55%	Only if GLM score > 3. Full policy + semantic analysis.	~800ms
 
Formula: raw = (rules × 0.15) + (glm × 0.30) + (qwen × 0.55)  →  adjusted = min(100, raw × session_factor)  →  final = apply_firewall_mode_cap(adjusted)
 
Action Thresholds

Score Range	Action	Behavior

0 – 29	ALLOW	Forward to LLM unchanged. Log green status.

30 – 54	FLAG	Forward but boost session_factor +0.15. Amber indicator on dashboard.

55 – 79	SANITIZE	Rewrite with Qwen removing malicious part. Forward sanitized version.

80 – 100	BLOCK	Do not forward. Return safe rejection message. Log full audit trail.
 
Session Factor Escalation (Slow-Burn Detection)

•	Each turn scoring > 30 adds +0.15 to session_factor

•	session_factor caps at 2.0 (doubles the effective score maximum)

•	Resets to 1.0 after 5 consecutive clean turns

•	Stored in Redis under sentinel:session:{id} with TTL of 3600s

Updated ThreatEvent Schema

Add these fields to the existing Pydantic ThreatEvent model:
 
class ThreatEvent(BaseModel):

    request_id: str

    timestamp: datetime

    session_id: str

    score: float                   # 0-100 final score

    action: str                    # ALLOW / FLAG / SANITIZE / BLOCK

    attack_type: str               # prompt_injection / jailbreak / pii / slow_burn / benign

    confidence: float              # 0.0 – 1.0 normalized from score

    reason: str                    # Human-readable explanation

    layer_triggered: str           # rule_engine / GLM / Qwen / Nemotron

    latency_ms: int                # Total pipeline ms

    session_factor: float          # Current multiplier

    firewall_mode: str             # strict / balanced / open

    user_trust_level: str          # new / regular / suspicious
 
Example Output

{

  "attack_type": "prompt_injection",

  "confidence": 0.91,

  "reason": "Instruction override detected — role persona replacement pattern",

  "layer_triggered": "GLM",

  "action": "BLOCK",

  "score": 92,

  "latency_ms": 218,

  "firewall_mode": "balanced",

  "user_trust_level": "suspicious"

}
 
Implementation Notes

•	The reason field is populated from the layer that had the highest contribution to the final score

•	confidence = score / 100 — keeps it simple, avoids over-engineering

•	layer_triggered = the deepest layer that ran (Qwen if escalated, GLM if not, rule_engine if short-circuited)

•	Dashboard shows attack_type, confidence %, reason, and layer_triggered on each threat card

NEW: User-Aware Defense

The system now tracks per-user trust levels and applies adaptive scoring thresholds. This makes Sentinel feel intelligent, not just reactive.
 
✅ NEW FEATURE — Beginner users get more lenient thresholds — Suspicious users get stricter ones
 
User Trust Levels

Trust Level	Criteria	Effect on Scoring	Threshold Adjustment

new	First 5 interactions	Lenient. Assume legitimate.	Block threshold raised to 90

regular	6–50 clean interactions	Normal. Default thresholds apply.	No change

flagged	2+ FLAG actions in session	Cautious. Qwen always runs.	GLM trigger lowered to 2

suspicious	1+ BLOCK in session or session_factor > 1.5	Strict. score × 1.2 bonus.	Block threshold lowered to 70
 
Redis Schema — User Profile

sentinel:user:{user_id}  →  {

  "trust_level": "regular",

  "total_turns": 24,

  "clean_turns": 22,

  "flag_count": 1,

  "block_count": 0,

  "last_seen": "2025-04-01T14:32:00Z"

}
 
Implementation Notes

•	user_id is extracted from the X-User-ID header (or falls back to session_id)

•	Trust level recalculates on every turn — no stale state

•	Dashboard shows each user's trust level badge in the session view

•	Q&A answer: 'We weight new users more leniently to avoid false positives during onboarding, and escalate automatically as suspicious patterns emerge — no manual configuration'

Critical Fixes Applied

1. spaCy Pre-Baked Into Docker Image

🔴 CRITICAL — Was: Runtime download (~800MB) — a build-time demo killer
 
Fix: Add the following line to the Dockerfile for the backend service. This caches the model in the image layer and eliminates the runtime download entirely.
 
# In Dockerfile — after pip install -r requirements.txt

RUN python -m spacy download en_core_web_lg
 
This means docker compose up --build takes longer on first build but the model is always available. Never download at runtime again.
 
2. GLM Timeout Contradiction Resolved

🔴 CRITICAL — Was: 2s hard timeout but 300ms average target — contradictory, judges will catch this
 
Fix: GLM hard timeout is now 300ms. If GLM does not respond in 300ms, the pipeline falls back to rule-engine-only scoring. The Q&A answer is: 'GLM averages 200ms. If it misses the 300ms deadline, we use the rule engine score alone and flag the request conservatively. Latency SLO is always met.'

•	Benign path latency: ~210ms (rule engine + GLM)

•	Escalated path latency: ~1.2s (rule + GLM + Qwen)

•	GLM-timeout fallback path: ~15ms (rule engine only, conservative action)
 
3. Nemotron Fail-Closed Behavior Defined

🔴 CRITICAL — Was: No fallback logic if Nemotron timed out or failed
 
Fix: Nemotron now fails closed. If Nemotron times out (>3s) or returns an error:

•	The LLM response is blocked — not passed through unscrubbed

•	ThreatEvent is logged with action: BLOCK, reason: 'Output auditor unavailable — fail-closed'

•	Dashboard shows a special amber indicator: 'Output audit failed — response withheld'

This is the safer default for a security product. Add this to Phase 5 error handling.
 
4. Authentication on Write Endpoints

🔴 CRITICAL — Was: All endpoints open — anyone could spam your OpenRouter key or add malicious policies
 
Fix: Add X-Sentinel-Admin-Key header check to all write endpoints. Single hardcoded value in .env.

# .env

SENTINEL_ADMIN_KEY=your-secret-key-here
 
# FastAPI dependency

def verify_admin(key: str = Header(alias='X-Sentinel-Admin-Key')):

    if key != os.getenv('SENTINEL_ADMIN_KEY'):

        raise HTTPException(status_code=403)
 
Apply to: POST /policies, POST /policies/{id}/toggle, POST /config (new). GET endpoints remain open for the demo dashboard.

8. Base64 Detection — Known Limitation, Practiced Answer

The rule engine handles base64 strings >20 chars and chained base64. It does not handle URL encoding or Unicode obfuscation beyond homoglyphs. This is fine for a hackathon. Have a practiced answer ready:

'Known limitation — our rule engine catches common obfuscation patterns. For production, we'd add a Unicode normalization pass and recursive URL-decode before the rule scan. We chose to scope this for the hackathon timeline.'
